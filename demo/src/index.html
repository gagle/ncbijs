How It Works

Problem: NCBI limits you to 3–10 requests per second. If you need millions of queries, that takes days.

Solution: Download the data once into your own database. Query it locally with the same @ncbijs/* packages. No rate limits.

1
Load
@ncbijs/etl downloads bulk NCBI data, parses it, and writes structured records to any storage backend (DuckDB, PostgreSQL, MotherDuck).
2
Query
The same @ncbijs/* packages query both sources. The only difference is where the data lives.
NCBI Servers
const datasets = new Datasets();
const genes = await datasets.geneBySymbol(
  ['TP53'], 'human'
);
Your Data (DuckDB)
const datasets = Datasets.fromStorage(db);
const genes = await datasets.geneBySymbol(
  ['TP53'], 'human'
);

The NCBI Servers panel queries NCBI APIs in real time using @ncbijs/mesh, @ncbijs/datasets, @ncbijs/clinvar, @ncbijs/pubchem, and @ncbijs/id-converter.

The Your Data (DuckDB) panel uses the exact same packages with .fromStorage() — pointed at a ~4 MB DuckDB database bundled with this demo as a sample. It contains a curated subset of real NCBI data (genes, MeSH descriptors, ClinVar variants, PubChem compounds, and ID mappings). Same methods, same return types, zero network requests.

In production, you'd run the ETL once (or on a schedule with @ncbijs/sync), then query your own copy — from a server, cloud database, or even the browser.
